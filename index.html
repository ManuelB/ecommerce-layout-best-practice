Layout best practices

Reusable concepts for best practice ecommerce layouting based on the latest and greatest web standards. For cheaper maintenance and better code.

General

The goal is to develop a layout philosophy which is reusable across brands, based on established standards, well documented and supported by Incentergy. It should be modular and updateable across the livetime of a project.

The basic idea of this methodology is to have a base styling which already covers the base cases and can be updated regularly. It must not be changed during a project. Requirements are:

  • Supported by 90% of the browsers
  • Documented
  • Responsive
  • Based on open standards
  • Optional: support high pixel density devices
  • Optional: consumable from a CDN
  • Optional: minimizable into one file
  • Optional: Should automatically generate sprites e.g. Sprite Factory
  • Optional: Eclipse support
  • Optional: Generate source maps
All enhancements and bug fixes from all projects should be pushed into the main repository and can be pulled

Base framework

To achieve the defined goals a base framework is necessary. This base framework should contain the building blocks for layouting the whole site.

Forms

All forms on web shop should look and behave the same. The styling should be compatible with the isinputfield module. So all forms should be supported by default without changing anything on the default HTML.

Tables

The base framework should contain default styles for tables to show them in a striped way and to highlight the currently hovered row. This makes the tables easier to read.

Business classes

Basis classes for indicating business events should be supported in paritcular:

  • Success e.g. applying a coupon, in stock
  • Information e.g. current promotion, approaching discounts
  • Warning e.g. coupon not active, product low on stock
  • Error e.g. wrong username, coupon not found, not in stock

Normalization

This part of the code should make sure that all browser behave in the same way by setting all the styles to a specified default. Typically normalize.css is used. They provide a test page.

Mixins for responsive grid

This is the most important part. The base framework should offer classes that make it possible to map the normal semantic elements and classes of a normal HTML page to different width on differnet device sizes. It is recommended to support 3 viewports desktop, tablet and mobile. Nevertheless the website should be fluid and resize according to width of the device.

Multiple device styles

It should be possible to restrict content to device type e.g. mobile, print and screenreader.

Accessibility

The framework should follow best practice accessibility guidelines according to W3C. An even more restricted list is given by the german BIENE award. In general using accessibility guidlines make maintenance a lot cheaper. Nice side effects are that the HTML5 is wellformed and therefore parseable by an ordinary XML parser. Testing tools that do browser automation like Selenium or CasperJS can use the site easily as well.

Non parts of base framework

It should not contain the following things:

  • Colors
  • Fonts
  • Absolute pixel measures for positioning, padding, margin or line-height
  • Dependencies to javascript

Multibrand environments

eCommrece systems are typically hosting multiple brands. In a perfect world the templates would stay consistent and only the CSS would be changed for a new brand. CSS Zen Gargen shows in an impressive way what is possible with semantic HTML code and smart CSS coding.

Workflow with designers

It is likely that there is an assigned person or an assigned team responsible for the design of one brand. The next section outlines the workflow how to collaborate with this person or team. The workflow supports and expects the following:

  1. The design is delivered in incremental steps
  2. The design is delivered as code e.g. SCSS
  3. The designer takes responsibility aboud the look and feel of the website
  4. The design team contains an expert in CSS
  5. The tech lead is able on a daily basis to integrate pull requests

Static interactive prototype approach

The image above shows that the designers are locally working with a static prototype. The static prototype is part of the base framework and should be in sync with the latest release of the ecommerce platform. That makes it possible to easily style new features and aftwards enable them on the storefront. Using an interactive prototype guarantees that there are only loosly coupled dependencies but that the produced deliverables are directly usable by the developers team.

The framework maintainer makes sure that the static prototype is always in sync with the most revent release

The designers can create spike solutions to build different campaigns for the business and the business can approve these solutions.

Process

The picture above shows how to handle the project design flow. The designers get an introduction into distributed version control and pull requests by the Teach Lead. Afterwards they clone the base framework repository from the official project repository. Now they can start to add designs and code CSS in the repository. All the time when they reached a certain milestone they can open a pull request to get a review for there code and integrate it into the main development repository.

Technical architecture

The designer can work in an incremental way. They can collaborate with each other and in the end will request the tech lead to integrate their designs into the system.

Updating parts of the layouts

The framework maintainer has the task to always care that the latest and greatest functionality is supported by the framework. Further he has to integrate bug fixes and features that are implemented during a project.

CSS Ratiocinator

This tools makes it possible to extract from a website all really used CSS rules by taking the rendered page and reverse engering the CSS that is necessary to display the DOM in the way it is displayed. This is especially helpful if CSS grew over a long period of time and the code should be reduced. Currently SCSS is supported as well by feeding the output through sass-convert

Implementation with ZURB 5

  • It is based on SASS
  • It is lightweight
  • It is well documented and accepted
  • It includes a grid system
  • It is responsive by default

Watching SCSS files

It is important that the precossor is fast to have short turn around cycles between changing scss files and seeing the result in the browser. The ruby command line tool sass can take up to 30s for bigger scss projects e.g. foundation. So it is recommended to use the fast c++ implementation libsass. It is hosted on github and the project website is libsass.org. The easiest way is to use the node-sass --watch option 

                    # on mac os x 10.10 including installed XCode 8.1, git and node
                    git clone https://github.com/sass/libsass.git
                    cd libsass
                    make
                    sudo make install
                    sudo npm -g install node-sass
                    node-sass --watch style.scss
